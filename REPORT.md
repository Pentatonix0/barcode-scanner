# Отчет по проекту Barcode Reader

## Суть приложения

Barcode Reader — это desktop/web-система для инвентаризации по штрихкодам. Оператор загружает Excel-каталог товаров, запускает сессию сканирования и получает поток сканов в реальном времени. Приложение считает количество и уникальные позиции, отдельно ведет нераспознанные штрихкоды, позволяет корректировать корзину вручную (decrement/delete/undo), сохраняет историю сессий в SQLite и формирует Excel-отчет по каждой завершенной сессии. Дополнительно реализованы SMTP-уведомления с отправкой отчета и настройка serial-порта сканера из UI.

## Технологии

### React 18

React использовался для построения интерфейса оператора: главный экран с KPI, панель настроек, история сессий и модальные окна подтверждения. Подход с компонентами и хуками упростил изоляцию доменной логики (например, `useSession`, `useCatalog`, `useSerialSettings`) и переиспользование UI-частей.

Краткое сравнение: Vue 3 дал бы похожий DX с Composition API, но в проекте уже используется React-экосистема, а TypeScript-типизация хуков и props здесь хорошо контролирует состояние сложного UI.

### TypeScript

TypeScript применялся для строгой типизации DTO, WebSocket-пэйлоадов, статусов сессии и форм настроек. Это снизило риск ошибок интеграции frontend ↔ backend и сделало рефакторинг безопаснее.

Краткое сравнение: чистый JavaScript быстрее на старте, но при росте количества API-методов и состояний увеличивает риск регрессий. TypeScript дает более предсказуемую поддержку кода.

### Vite

Vite использовался как dev/build-инструмент frontend: быстрый HMR в разработке и компактная production-сборка. Для этого проекта важна быстрая итерация интерфейса, и Vite дает минимальную задержку обратной связи.

Краткое сравнение: webpack гибкий, но тяжелее в базовой настройке и обычно медленнее в dev-цикле. Для текущего масштаба Vite практичнее.

### FastAPI

FastAPI выбран как HTTP/WebSocket-платформа backend. Он используется для REST-эндпоинтов (`/catalog`, `/session`, `/notifications`, `/serial`, `/system-settings`) и WebSocket-канала `/ws` для push-событий сканирования.

Краткое сравнение: Flask проще как микро-фреймворк, но FastAPI из коробки дает строгую валидацию и auto-docs через Pydantic, что удобнее для API-интеграций.

### Pydantic

Pydantic применяется для валидации DTO запросов/ответов и конфигурации через `BaseSettings`. За счет этого настройки (serial, smtp, reports) и API-контракты централизованно и типобезопасно описаны.

Краткое сравнение: marshmallow и ручная валидация требуют больше шаблонного кода. Pydantic лучше сочетается с FastAPI и ускоряет разработку.

### SQLAlchemy + SQLite

SQLite используется как встраиваемое хранилище сессий, позиций и системных настроек; SQLAlchemy (async) — как ORM/DAO-слой. Для офлайн desktop-приложения это минимальная по операционным затратам связка.

Краткое сравнение: PostgreSQL мощнее для многопользовательской нагрузки и сложной аналитики, но требует отдельного сервера. Для локального single-user сценария SQLite рациональнее.

### openpyxl

openpyxl применяется в двух местах: загрузка каталога товаров из `.xlsx` и экспорт отчета завершенной сессии обратно в `.xlsx` (листы `Scanned Items` и `Unknown Items`).

Краткое сравнение: pandas удобен для аналитики и ETL, но здесь не нужны тяжелые табличные операции. openpyxl легче и лучше подходит под прикладной I/O Excel-файлов.

### pyserial

pyserial отвечает за интеграцию с физическим scanner-устройством через COM/tty порт, чтение баркодов и реконнект при ошибках порта. Это ядро real-world сценария проекта.

Краткое сравнение: прямые OS-специфичные API сложнее сопровождать кроссплатформенно. pyserial дает единый интерфейс и быстрее внедряется.

### WebSocket

WebSocket нужен для немедленной доставки событий сканирования из backend в UI (новый скан, неизвестный баркод, undo, изменение количества). Это обеспечивает «живой» интерфейс без опроса REST по таймеру.

Краткое сравнение: polling проще, но создает задержки и лишний трафик. Для событийной модели сканера WebSocket функционально правильнее.

### Electron

Electron используется как desktop-контейнер: поднимает backend-процесс, ждет `/health`, передает runtime-конфиг в renderer и добавляет desktop-функции (например, открыть папку с отчетом).

Краткое сравнение: Tauri обычно легче по размеру приложения, но Electron дает зрелую экосистему и предсказуемое поведение для связки Node + локальный backend-процесс.

## Заключение

Проект показал практичную архитектуру для офлайн-сканирования: React-интерфейс + FastAPI-сервис + локальная БД + Excel-интеграция. Наибольшую ценность дали строгие контракты (TypeScript/Pydantic), событийная модель (WebSocket) и настройка инфраструктуры через UI (serial/smtp/reports).

Из ограничений: зависимость от физического сканера и окружения затрудняет автоматизированные e2e-проверки UI. Следующим шагом разумно добавить слой эмуляции сканера для автотестов и формализовать smoke-сценарии упаковки desktop-сборки.
